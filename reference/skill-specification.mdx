- Read [Epic Specification](/epic-specification) when you need the normative

## What an epic is

An epic is a valid `SKILL.md` package plus an `EPIC.md` file and, optionally,
an operating file set around it.

```directory
my-epic/
├── SKILL.md
├── EPIC.md
├── runtime/
│   ├── plans/
│   ├── state.json
│   ├── log/
│   ├── ROADMAP.md
│   ├── DECISIONS.md
│   └── artifacts/
├── hooks/
├── cron.d/
├── policy.yml
├── skills/
├── scripts/
├── references/
└── assets/
```

The key distinction is that the epic can hold durable state and operating
behavior, not just reusable instructions.

## Required files

Every epic requires:

- `SKILL.md` for discovery, activation, and the Agent Epics gateway footer
- `EPIC.md` for durable identity and operating definition

`SKILL.md` keeps the package compatible with existing skill loaders and, in
`0.5.2`, also gives first-contact orientation into the epic system.
Published epics may also include an optional `metadata.source` hint so hosts
can try to locate the full epic when only `SKILL.md` is available.
`EPIC.md` is what makes the package an epic.

## `EPIC.md`

`EPIC.md` contains:

- YAML frontmatter with required identity fields
- a Markdown body describing the epic's purpose, scope, phases, success model,
  and resumption guidance

Required frontmatter fields:

| Field | Purpose |
|-------|---------|
| `id` | Unique epic identifier within the workspace |
| `spec_version` | EPIC spec version targeted by the file |

Common optional fields:

| Field | Purpose |
|-------|---------|
| `tags` | Lightweight classification tags |
| `timezone` | Default timezone for runtime behavior |

The body should describe what stays true across sessions and, for reusable
workflow epics, across runs.

## Core operating files

The core operating surface is usually built from these files:

| File or directory | Role |
|-------------------|------|
| `runtime/plans/` | Tactical working loop: now, next, blocked |
| `runtime/state.json` or `runtime/state/` | Machine-readable facts and lifecycle state |
| `runtime/log/` | Append-only durable history |
| `runtime/ROADMAP.md` | Longer-horizon milestones for the current run |
| `runtime/DECISIONS.md` | Curated decision memory for the current run |
| `runtime/artifacts/` | Outputs, deliverables, and generated work |

An epic does not need all of them to be valid. They appear as the epic is used.

## State precedence

When both state forms exist:

- `runtime/state/core.json` is authoritative over `runtime/state.json`

Reserved state fields:

| Field | Meaning |
|-------|---------|
| `state_version` | Schema version for the structured state |
| `status` | Lifecycle status: `active`, `paused`, `complete`, `abandoned` |
| `current_plan` | Active plan filename in `runtime/plans/` |

Everything else in structured state is epic-defined.

## Logs

Logs provide durable continuity across sessions, collaborators, and autonomous
runs.

Filename format:

`YYYY-MM-DDTHH-MM-SSZ-{actor}-{slug}.md`

Logs are append-only except for redaction.

## Executor surface

Executor extensions let an epic operate beyond interactive resumption:

| Feature | Purpose |
|---------|---------|
| `hooks/` | Event-triggered reactions |
| `cron.d/` | Recurring execution |
| `policy.yml` | Hard constraints on autonomous operation |

These features are specified in [Epic Executor Specification](/epic-executor). A compliant
executor must either honor them or surface that it does not support them.

## Archetypes

Most epics fall into one of three patterns:

- **Workflow epic**: reusable operating model with fresh state for each run
- **Instance epic**: one specific long-lived project
- **Capability epic**: installed agent capability that keeps operating over time

See [Epic archetypes](/epic-creation/archetypes) for concrete examples.

## Compatibility model

Every epic is still a valid skill package. That means:

- skill-aware systems can still discover `SKILL.md`
- epic-aware systems can additionally detect `EPIC.md`
- `0.5.2` hosts can use the standardized `## Agent Epics` footer as a portable compatibility bridge
- the surrounding operating files provide durable state, memory, and runtime behavior

If you need the underlying `SKILL.md` compatibility rules, read the
[SKILL.md Reference](/skill-specification).
